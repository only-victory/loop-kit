# /proposal : 제안서 제작

수주·승인·협업을 끌어내는 **외부 의존성 0개의 단일 HTML 제안서**를 만든다. 파일 하나만 보내면 받는 사람이 어떤 환경에서든 열고, 그대로 인쇄(PDF)한다.

> 이 스킬의 목표는 "우리를 소개하는 것"이 아니라 **"상대가 '이건 우리 문제를 푸는구나'라고 느끼고 다음 미팅을 잡게 하는 것"**이다.

## 제1원칙 : 상대의 문제가 먼저다

제안서는 우리 자랑이 아니다. **첫 화면에 "당신의 이 문제"가 보여야 한다.** 회사 소개·연혁·수상 이력은 뒤다. 상대가 "내 문제를 정확히 안다"고 느껴야 그다음 우리 솔루션을 듣는다. 솔루션은 항상 문제와 1:1로 대응시킨다.

## 절차 (이 순서를 지킨다)

1. **인터뷰** : 아래 [깊은 인터뷰]로 결·상대·핵심을 끌어낸다.
2. **한 줄 제안** : 제안서를 쓰기 전에 "우리는 [상대]의 [문제]를 [방법]으로 푼다"를 한 줄로 쓴다.
3. **구조 설계** : 고른 결의 골격에 섹션을 배치한다 (아래 [제안 결]).
4. **제작** : 단일 HTML로 만든다. 문제 → 솔루션 → 근거 → 일정 → 견적 → 다음 단계 흐름.
5. **검증** : `scripts/verify.sh`로 규약·필수 섹션을 확인하고, [육안 체크리스트]로 설득력을 본다.

## 깊은 인터뷰 (옵션은 고르게, 내용은 받는다)

선택지가 있는 건 **선택 창**으로 띄워 고르게 한다. 직접 받는 건 **실제 제안 내용**이다.

**0단계 : 제안 결 + 밀도 (선택 창으로 고르게 한다)**

- **제안 결** : 목적에 맞는 것을 고른다
  - ① B2B 수주결 : 고객 문제 → 솔루션 → 차별점 → 견적 → 레퍼런스 (외부 고객 설득)
  - ② 내부 승인결 : 현황·문제 → 해결안 → 기대효과·ROI → 필요 자원 → 리스크 (의사결정자 승인)
  - ③ 파트너십결 : 공동 기회 → 역할 분담 → 상호 이익 → 협업 모델 → 다음 단계 (대등한 협업)
- **밀도** : 압축(1\~2장) / 표준(3\~5장) / 상세(6장+, 근거·부록 포함)

**1단계 : 상대·목적 (여기서 깊이가 갈린다)**

- **받는 사람** : 실무 담당(상세·근거) / 결정권자(결론·ROI) / 혼합(요약+상세)
- **상대가 지금 겪는 문제** : 가장 아픈 것 1\~3개 (제안서 전체가 이걸 푸는 구조)
- 이 제안서를 받고 상대가 **무엇을 하길** 바라나 (마지막 CTA를 정한다)

**2단계 : 핵심 내용 (필수)**

- **우리 솔루션** : 문제마다 1:1로 대응하는 해결책
- **차별점·근거** : 왜 우리인가 (레퍼런스·실적·수치 3개 이내)
- **일정·견적** : 추진 일정과 투자 규모 (범위·전제 명시)

**3단계 : 상세 내용**

- 추진 전략·방법론, 투입 인력·역할, 기대효과(정량/정성), 리스크·대응, 회사 소개·실적

## 제안 결 (결마다 섹션 순서가 다름, 기술 규약은 공통)

| 결 | 섹션 골격 (위→아래) |
|----|---------------------|
| B2B 수주결 | 한 줄 제안 → **고객 문제** → 제안 솔루션(문제 1:1) → 차별점·레퍼런스 → 추진 일정 → 견적 → 다음 단계 |
| 내부 승인결 | 한 줄 제안 → **현황·문제** → 해결안 → 기대효과·ROI → 필요 자원·일정 → 리스크 → 승인 요청 |
| 파트너십결 | 한 줄 제안 → 공동 기회 → 역할 분담 → 상호 이익 → 협업 모델·일정 → 다음 단계 |

**공통 기술 규약:**
- 외부 CDN script 0개, 외부 CSS 0개 (자체완결), CSS는 `<style>` 인라인
- `@media print` 인쇄 대응 필수 (제안서는 인쇄·PDF로 자주 본다)
- 제목 `<h1>` 필수 (제목은 회사명이 아니라 한 줄 제안이 된다)
- em-dash·en-dash 금지, 구분자는 콜론(`:`)

**컴포넌트 클래스:**

| 클래스 | 용도 |
|--------|------|
| `.proposal-summary` | 한 줄 제안 + 핵심 요약 (첫 화면) |
| `.problem-box` | 고객 문제·현황 (가장 먼저, 강조) |
| `.solution-card` | 솔루션 (문제와 1:1 대응) |
| `.kpi-row` | 기대효과·ROI 수치 강조 |
| `.timeline` | 추진 일정·마일스톤 |
| `.quote-table` | 견적·투자 표 |
| `.ref-box` | 레퍼런스·실적 |
| `.cta-box` | 다음 단계·연락처 (마지막) |

## 안티패턴 (이게 보이면 다시 만든다)

- 첫 장이 회사 소개·연혁 → 상대 문제가 먼저다. 회사 소개는 뒤로.
- 솔루션이 문제와 따로 놂 → 솔루션은 문제마다 1:1로 붙인다.
- 제목이 "OO 제안서" → "OO의 이탈률을 6개월 만에 절반으로"처럼 제안이 제목이다.
- 견적에 전제·범위가 없음 → 숫자만 있으면 협상에서 흔들린다.
- 마지막이 "감사합니다"로 끝 → 마지막은 구체적 다음 단계(미팅 날짜·연락처)다.
- 결정권자용인데 5장 내내 방법론 → 결정권자는 결론·ROI만 본다. 결을 맞게 골라야 한다.

## 검증 (필수 : loop-kit 게이트)

```bash
bash ${CLAUDE_SKILL_DIR}/scripts/verify.sh <만든-제안서.html>
```
