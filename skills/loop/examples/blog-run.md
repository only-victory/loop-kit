# 블로그 루프 : 1회 실행 흔적

`reference/blog.md`를 실제로 한 번 돌린 기록. 사람이 멈추는 두 지점(검수·승인)이 어떻게 보이는지 보여준다.

## 트리거

매일 07:00 스케줄 (이날은 수동 호출).

## 소스 (기계)

주제 영역 "AI 업무 자동화" + 키워드로 서치 어댑터 실행. 후보 5건 수집.

```
1. (출처 A) 새 코딩 에이전트 공개          [당일]
2. (출처 B) 사내 AI 도입 설문 결과         [3일 전]
3. (출처 C) 프롬프트 작성법 정리글          [당일]
4. (출처 D) AI 규제 입법 동향              [당일]
5. (출처 E) 어제 쓴 글과 같은 주제          [당일, 중복]
```

게이트 : 5건 모두 출처 URL 있음. 단 2번은 당일치 아님, 5번은 중복.

## 검수 (★ 사람 정지)

> 루프가 멈추고 묻는다 : **후보 5건 중 무엇을 채택할까?**

사람 판단 :
- 2번 제외 (당일치 아님), 5번 제외 (어제와 중복).
- 1번 채택, 각도는 "도구 소개가 아니라 우리 팀 워크플로에 어떻게 끼우나".

출력 : 채택 1건 + 각도 확정.

## 변환 (기계)

채택 주제 + 각도로 blog 스킬 호출. 제목·소제목·메타가 갖춰진 마크다운 초안 나옴. 게이트 : blog 게이트(H1·H2·메타·분량) 통과.

## 다듬기 (기계)

humanize 스킬 호출. 의례적 도입부·빈 수식어·기계적 리듬 제거. humanize 게이트 통과.

## 검증 (기계)

발행 규약 루브릭 점검 : 제목 있음, 길이 적정, 링크 유효. → 합격.

## 승인 (★ 사람 정지)

> 루프가 멈추고 묻는다 : **이 글, 내 이름 달고 발행할까?**

사람 통독 : 한 문단이 설명 과잉이라 반려. 그 문단만 줄여 재검증 → 합격 → 발행 확정.

## 싱크 (기계)

게시 어댑터 실행 (repo 마크다운 커밋). 발행 URL 확인.

## 회고

- 사람이 개입한 건 딱 두 번, 둘 다 **판단**이었다 (무엇을 쓸지 / 내보낼지). 서치·생성·다듬기·게시는 손 안 댐.
- 검수에서 5건을 1건으로 줄인 게 글 품질을 정했다. 이 단계를 자동화했으면 평범한 글이 매일 발행됐을 것이다.
- 변환 단계는 이제 blog 스킬이 맡는다. 처음 이 루프를 정의할 땐 글 생성 스킬이 없어 프롬프트 단계로 비워뒀는데, 그 공백이 blog 스킬을 끌어냈다. 루프를 먼저 그리면 다음에 만들 스킬이 보인다.
