# 블로그 루프 : 범용 첫 구현

매일/매주 같은 흐름으로 블로그 한 편을 뽑는 파이프라인. 소스·싱크는 어댑터라 어떤 서치 도구·게시 대상에도 꽂힌다.

## 0. 메타

- **루프 이름** : blog
- **트리거** : 스케줄 (예 : 매일 07:00) 또는 수동 호출
- **공개 범위** : 외부 (내 이름 달고 발행 → 승인 게이트 필수)

## 1. 단계 계약표

| 단계 | 입력계약 | 처리 | 출력계약 | 게이트 | 담당 |
|------|----------|------|----------|--------|------|
| 소스 | 주제 영역 + 키워드 | 핫이슈 서치 (어댑터) | 후보 글감 N건 + 출처 URL | 출처 있음 · 당일치 자료 | 기계 |
| 검수 | 후보 글감 N건 | 진위 · 중복 · 적합성 판정 | 채택 1~2건 + 다룰 각도 | 중복 아님 · 내 주제 영역 | 사람 |
| 변환 | 채택 주제 + 각도 | blog 스킬 호출 (초안 생성) | 마크다운 초안 (제목·소제목·메타) | blog 게이트 통과 (H1·H2·메타·분량) | 기계 |
| 다듬기 | 마크다운 초안 | humanize 스킬 호출 | 슬롭 제거된 글 | humanize 게이트 통과 | 기계 |
| 검증 | 다듬은 글 | 발행 규약 루브릭 점검 | 합격 또는 불합격 사유 | 제목 · 길이 · 링크 규약 충족 | 기계 |
| 승인 | 합격한 글 | 사람이 최종 통독 | 발행 확정 또는 반려 | 내 이름 달 만한 품질 | 사람 |
| 싱크 | 승인된 글 | 게시 (어댑터) | 발행 URL | 발행 확인됨 | 기계 |

## 2. 어댑터 (환경에 꽂는 부분)

루프 정의엔 박지 않는다. 실제 환경에서 연결한다.
- **소스 어댑터** : 웹 서치 / 뉴스 API / RSS 중 택1. 출력은 `후보 글감 + 출처 URL`로 통일.
- **싱크 어댑터** : repo 마크다운 커밋 / CMS API / 정적 사이트 중 택1. 입력은 `승인된 마크다운`으로 통일.

어댑터의 출력·입력 형태만 계약과 맞으면 내부 단계는 그대로 돌아간다.

## 3. 공백이 메워진 과정 (루프가 스킬을 끌어냈다)

- **변환 단계의 글 생성 스킬이 처음엔 없었다.** 이 루프를 처음 정의할 때 loop-kit엔 humanize(다듬기)만 있고 글을 **생성**하는 스킬이 없어, 변환 단계를 "프롬프트 단계"로 비워뒀다.
- **그 공백이 `blog` 스킬을 끌어냈다.** 루프를 먼저 정의하니 "다음에 만들 스킬"이 드러났고, 이후 `blog` 스킬이 추가되어 지금은 변환 단계가 `blog`를 직접 호출한다. 이것이 루프의 효용이다 : 무엇을 자동화할지가 아니라 **무엇을 다음에 만들지**를 보여준다.

## 4. 체크포인트 (사람이 멈추는 곳)

- **검수** : 서치 결과가 쏟아져도 1~2건만 고른다. 여기서 안 거르면 평범한 글이 양산된다.
- **승인** : 검증을 통과해도 통독 없이 발행하지 않는다. 외부 공개물의 마지막 방어선.

내부용 루프(슬랙 다이제스트 등)라면 승인 게이트를 낮추거나 생략할 수 있다. 블로그는 외부라 유지한다.
