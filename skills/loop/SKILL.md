# /loop : 작업 사이클 오케스트레이션

스킬이 **동사**(다듬다·정리하다·만들다)라면, 루프는 **문장**이다. 소스에서 가져와서 → 스킬로 변환·검증하고 → 사람이 판단한 뒤 → 내보내는, **반복 가능한 파이프라인** 하나를 정의하고 돌린다.

이 스킬은 **자기만의 산출물 형식을 만들지 않는다.** 마지막 단계 스킬이 내는 것을 그대로 내보낸다 (블로그 글은 humanize 산출, 월간 보고 덱은 slide 산출). 그래서 기존 스킬들의 산출물 형태를 손대지 않고 그 위에 얹힌다.

## 루프의 6단 해부 (계약)

모든 루프는 각 단계를 **계약**으로 못박는다. 모호하면 자동화도 위임도 교육도 안 된다.

| 단계 | 무엇 |
|------|------|
| **소스** | 어디서 가져오나 (서치·슬랙·Gmail·RSS). **어댑터로 분리** = 환경마다 다른 부분 |
| **검수** | 가져온 게 할 가치 있나 (진위·중복·적합성). 보통 **사람** |
| **변환** | 어떤 스킬을 어떤 순서로 (예: 초안 → humanize) |
| **검증** | 각 단계 출력이 게이트를 통과하나 (verify.sh·루브릭) |
| **승인** | 내 이름 달고 나갈 만한가. 외부 공개물이면 **반드시 사람** |
| **싱크** | 어디로 내보내나 (게시·슬랙·파일). **어댑터로 분리** |

각 단계는 `입력계약 → 처리 → 출력계약 → 게이트 → 담당(기계/사람)`을 모두 명시한다. 하나라도 비면 그 루프는 미완성이다.

## 사람이 남는 자리 (핵심 원칙)

베스트 프랙티스는 전부 자동이 아니다. **사람은 노동이 아니라 판단 지점에 남는다.**
- 사람 : 검수(할 가치 있나) · 승인(내 이름 걸 만한가)
- 기계 : 서치 · 포맷 · 변환 · 게시

이 구분을 놓치면 자동화가 아니라 **슬롭을 스케일**하는 것이다. 내부용(슬랙 다이제스트)은 게이트를 낮춰도 되지만, **외부 공개물(블로그·발표)은 승인 게이트가 필수**다.

## 절차

1. **결정** : 어떤 루프인가. 기존 정의가 있으면 `reference/<루프>.md`를 연다 (예: `blog.md`).
2. **새 루프면** : `reference/anatomy.md`(빈칸 스켈레톤)를 복제해 6단 계약을 채운다.
3. **어댑터 확인** : 소스·싱크에 이 환경의 실제 도구를 꽂는다 (서치 도구·게시 대상). 템플릿엔 박지 않는다.
4. **실행** : 단계 순서대로 진행한다. 변환 단계는 해당 스킬(blog·proposal·report·slide·humanize·proofread 등)을 호출한다.
5. **체크포인트 정지** : 검수·승인 단계에서 **반드시 멈추고 사람에게 묻는다.** 외부 공개물은 승인 없이 싱크로 넘어가지 않는다.
6. **그 외엔 묻지 않는다** : 검수·승인 **외의 단계(소스·변환·검증·싱크)에서는 절대 사람에게 묻지 않는다.** 기본값으로 자동 진행하고, 입력이 모호하거나 막히면 멈추지 말고 합리적 기본값으로 처리한 뒤 그 가정을 검수 시점에 함께 보고한다. 사람이 손대는 곳은 검수·승인 두 곳뿐이다.

## 검증 (필수 : loop-kit 게이트)

루프 **정의**도 검증한다. "검증 없으면 예쁜 쓰레기"는 루프 자체에도 적용된다.

```bash
bash scripts/verify.sh reference/blog.md
# 루프 단계 : N개
# ✓ HARD 위반 0건 : loop 통과
```

린터가 막는 것 : 빈/미작성 계약(`___`·TODO), 6컬럼 누락, **사람 체크포인트 0개**, em/en-dash. 통과해야 "정의가 완성된 루프"다.

## 참조

- 빈칸 스켈레톤(범용 문법, 새 루프 복제 원본) : `reference/anatomy.md`
- 블로그 루프 첫 구현 : `reference/blog.md`
- 실제 1회 실행 예시(체크포인트 포함) : `examples/blog-run.md`

새 루프(경쟁사·슬랙)는 `anatomy.md`를 복제해 `reference/`에 한 장씩 추가하면 된다. 스킬은 계속 `loop` 하나다.
