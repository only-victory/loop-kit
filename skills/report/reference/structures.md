# 보고서 구조 : 결론 먼저, 종류별 골격

보고서는 "읽는 사람의 시간을 아끼는 문서"다. 가장 중요한 원칙은 **결론을 맨 앞에**. 바쁜 독자는 요약만 읽고 의사결정한다.

## 제1원칙 : 역피라미드

```
결론/요약  →  근거  →  상세  →  부록
(맨 위에 답)   (왜)    (어떻게)  (데이터)
```

서론부터 빌드업하지 않는다. **첫 화면에서 "그래서 무엇"이 보여야** 한다.

## 종류별 골격

| 종류 | 첫 섹션 | 핵심 | 마무리 |
|------|---------|------|--------|
| **분석** | 핵심 발견 1줄 | 데이터 → 해석 → 함의 | 권고 |
| **현황** | 한 줄 상태(정상/주의/위험) | 지표 + 변화 + 원인 | 다음 액션 |
| **조사** | 결론 답 | 질문 → 방법 → 근거 | 한계·후속 |
| **제안** | 제안 한 문장 | 문제 → 해결 → 기대효과 → 비용 | 의사결정 요청 |

## 독자별 깊이

| 독자 | 무엇을 원하나 | 조정 |
|------|---------------|------|
| **경영진** | 결론·의사결정 | 요약 1페이지, 수치는 핵심만, 부록에 상세 |
| **실무** | 근거·재현 | 방법·데이터·가정 충실히 |
| **외부** | 맥락 포함 | 전제·용어 풀이 추가 |

## 수치를 다루는 법

- 모든 수치에 **출처**를 단다 (N수·기간·조사방법)
- 단정 전에 **한계**를 밝힌다 (단일 분기·표본 편향 등)
- 비교는 기준선과 함께 ("23% 증가" → "전분기 대비 23% 증가")

## 자체완결 원칙 (loop-kit)

외부 폰트·CDN·스크립트를 하나라도 참조하면 그 링크가 죽는 순간 보고서가 깨진다.
**CSS는 `<style>`에 인라인**, 차트가 필요하면 표·인라인 SVG로. 그래야 이메일 첨부·오프라인·아카이브 어디서든 그대로 열린다.
